Chat App Prototype Mockup

Chats

What: The chats have a randomly generated channel id, as well as a title, which are data that need to be persisted and shared. There are also additional graffiti objects to allow joiners to be detected and see team data.

Where:Anyone using the app can discover any of the teams, as long as they've joined that team. So, we've used a hardcoded channel "chappystick-v1" for the where.

Who: Only people that made a chat can see it. So, we make an allowed list that starts with just the creator id.

Bookmarks

What: These are the names for chats that the user joins, rather than creates. They essentially have the same data as chats, with channel and title. They are eventually merged with chats to represent all the chats the user is in.

Where:For the same reason as above, these use the "chappystick-v1" channel.

Who: People can see this when they join the team, so they are put in the allowed list.

Messages

What:These messages have a content attribute as well as a published date.

Where:These are in a single current channel, so they are in channel.value.

Who: Anyone in the channel can see the message, so no allowed list needed.

Meetings

What: Meetings need name, time, date, and location data to be useful.

Where:Meetings are seen in their associated current channel, so they are in channel.value.

Who: People can see the meetings when if they are in the channel, so no allow list is needed.

Other design decisions

While this is a small mockup of the chat app design, I still made sure to follow design principles. I included a copy button for the team id/join code for efficiency, and made the meeting time to autofill to 7pm for the current day to hopefully reduce the keystrokes needed to schedule a meeting. This could however be more optimized with user testing. For safety, I included the ability to leave a group. For learnability and efficiency, I made the nearest meetings in the meetings header sort to the top so that new users can immediately see the most important information.
